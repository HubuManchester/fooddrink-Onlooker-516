# 英文配音稿（TTS 朗读用）

> 把下面的英文复制到文字转语音工具中，逐段生成配音，配合视频画面使用。


## 0. Opening

Hi, I'm Yang Senyu. This is my Mobile Computing coursework project — FoodPicker, a cross-platform app that helps you decide what to eat. The app is built with .NET MAUI, using XAML for the UI, C-sharp for the logic, and following the MVVM architectural pattern. I will now walk through each of the seven marking criteria one by one.


## 1. UI/UX Design and Accessibility

The app has four main pages: Discover for random recommendations, Compass for direction-based picks, Album for food photos, and Favourites for saved dishes. The bottom tab bar provides clear, intuitive navigation.

All pages use MAUI's AppThemeBinding for automatic dark mode adaptation. I defined eighteen custom theme colour variables in Colors.xaml. When the system switches to dark mode, every page adapts seamlessly. Backgrounds shift from warm white to deep brown. Text changes from dark grey to warm white. This aligns with WCAG 1.4.3 contrast requirements.

All interactive elements include SemanticProperties dot Description and Hint for screen reader support. Buttons maintain a minimum forty-four by forty-four pixel touch target, meeting WCAG 2.5.5 target size guidelines. Every input field has a clear placeholder hint.


## 2. Mobile Hardware

First, the accelerometer. On the Discover page, shaking the phone triggers the Accelerometer dot ShakeDetected event. I added an eight-hundred-millisecond debounce to prevent duplicate triggers. Each shake randomly picks a new dish, accompanied by a custom rattle sound effect and card bounce animation.

Second, the compass and magnetometer. The Compass page uses the Compass API to read magnetometer data. I implemented exponential smoothing to reduce sensor noise. Each new reading contributes only twelve percent, blended with the previous frame. Changes under two degrees are ignored entirely. The rotation uses CubicOut easing with a SpringOut bounce-back for smooth, natural movement. The compass recommends dishes by direction. Facing north gives northern snacks, south for Cantonese, southwest for Sichuan.

Third, the camera. The Album page uses MediaPicker dot CapturePhotoAsync to invoke the system camera. Camera permission is requested and checked before capture. After taking a photo, a category picker appears so users can tag their photos. Photos are saved to the apps local directory and displayed in a two-column grid with category badges.

Fourth, geolocation. Geolocation uses the Geolocation API to obtain GPS coordinates, then Geocoding API for reverse geocoding. The code handles multiple error scenarios: permission denied, feature not supported, network timeout. All with user-friendly messages rather than raw exceptions.

Fifth, haptic feedback. Haptic feedback cannot be felt in the emulator, but on a physical device every button tap produces a short vibration. Shaking triggers a longer one. Looking at the code, every button command and shake handler calls HapticFeedback dot Default dot Perform. Short taps use the Click type, shakes use LongPress. Each call is wrapped in try-catch so unsupported devices never crash.


## 3. Functionality

Now I will demonstrate the complete user flow. On Discover, tapping Pick Another randomly changes the dish with a bounce animation. Tap Save, and the button turns grey showing Saved. Type a keyword like noodle and hit Search to find a match from fifty-three dishes. Tap Share to invoke the system share sheet and send a recommendation to friends.

On Favourites, a stats card at the top shows the total saved dishes and your top category. Tap a saved dish to write a personal note. A badge appears on the card when a note is saved. Swipe left to delete with a confirmation dialog to prevent accidents. Tap Pick One for Me to randomly select from your favourites. Clear All at the bottom empties everything.

The Compass page rotates with your phone, recommending region-specific dishes in real time as you turn. The Album page supports both camera capture and gallery import, with category filter chips at the top.


## 4. Validation and Error Handling

Leave the search box empty and tap Search. An alert appears: Please enter a food name. Type just one letter. The alert says: Name is too short. At least two characters. Enter valid text, and the search works normally.

The validation logic lives in the ValidateCraving method. It returns a tuple with an isValid boolean and a message string. The empty check uses string dot IsNullOrWhiteSpace. The length check uses Length less than two. All error messages are user-friendly natural language.

Every hardware call is wrapped in try-catch. Compass, camera, geolocation, haptic feedback, sound effects. No single feature failure can crash the app. Permission denials give clear guidance. Unsupported features degrade gracefully.


## 5. Code Quality

The project follows standard MVVM architecture. Models, ViewModels, Views, Services, Helpers. Clear separation of concerns.

FavoritesService uses the singleton pattern. All pages share the same favourites data through a FavoritesChanged event, so ViewModels never need to poll. Private fields use underscore camelCase prefix. Public properties use PascalCase. Methods use verb-first naming. Every class and method has XML documentation comments.

Nine custom converters are reused across multiple pages. InvertedBool, IsEmpty, IsNotEmpty, BoolToColor, FavBtnColor. All registered globally in App dot xaml's ResourceDictionary.


## 6. Cross-Platform Deployment

The app is currently running on my Android phone. The exact same codebase, with zero modifications, compiles and runs directly on Windows. All four tabs, dark mode, every feature, identical across platforms. This demonstrates dot NET MAUI's true cross-platform capability. Write once, run everywhere.


## 7. GitHub Usage

This is my GitHub repository. The entire development process spans thirty-three commits. From initial project setup, progressively adding hardware features, refining the UI, fixing bugs, to final polish. Each commit has a clear descriptive message. The README includes project overview, feature list, hardware inventory, tech stack, and project structure.


## 8. Closing

That concludes my FoodPicker project walkthrough. It covers five mobile hardware features, four fully functional pages, MVVM architecture, dark mode and accessibility support, input validation and error handling, and cross-platform deployment. Thank you for watching.
